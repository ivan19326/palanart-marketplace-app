# Паланарт: внешний auth-слой для VK ID и Telegram

## Зачем это нужно

GitHub Pages отдает только статический фронтенд. Для безопасного OAuth через `VK ID` и `Telegram` нужен внешний слой, который:

- хранит client secret вне браузера;
- принимает callback от провайдера;
- обменивает code на токен;
- получает профиль пользователя;
- создает или находит пользователя в `Supabase`;
- возвращает пользователя обратно на сайт `Паланарт`.

## Рекомендуемая схема

1. Пользователь на сайте нажимает `Войти через VK ID` или `Войти через Telegram`.
2. Фронтенд открывает внешний URL авторизации, который задается в админке:
   - `VK auth URL`
   - `Telegram auth URL`
3. Внешний auth-сервис принимает запрос и знает:
   - роль пользователя (`client` или `artist`);
   - `returnTo` — страницу возврата на сайт.
4. Сервис делает OAuth у провайдера.
5. После успешного входа сервис:
   - получает профиль пользователя;
   - ищет или создает пользователя в `Supabase Auth`;
   - записывает служебный профиль в таблицы маркетплейса;
   - выдает сессию;
   - редиректит обратно на `returnTo`.

## Минимальный стек

- `Cloudflare Workers` или `Supabase Edge Functions` для callback и обмена кодов;
- `Supabase Auth` как единая точка хранения пользователей;
- отдельные переменные окружения:
  - `VK_CLIENT_ID`
  - `VK_CLIENT_SECRET`
  - `TELEGRAM_BOT_TOKEN` или параметры Login Widget
  - `SUPABASE_SERVICE_ROLE_KEY`

## Что уже готово в сайте

- кнопки `Google`, `VK ID`, `Telegram` в интерфейсе;
- режим `Supabase` по умолчанию;
- внешние URL для `VK` и `Telegram` можно задать в админке;
- фронтенд передает в внешний auth URL:
  - `role`
  - `returnTo`

## Что сделать следующим этапом

1. Поднять внешний auth endpoint для `VK ID`.
2. Поднять внешний auth endpoint для `Telegram`.
3. Подключить серверную выдачу сессии `Supabase`.
4. Прогнать сценарий:
   - клиент -> соцвход -> кабинет;
   - исполнитель -> соцвход -> кабинет;
   - повторный вход существующего пользователя.
